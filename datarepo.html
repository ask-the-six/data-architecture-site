Data Repository Concept

ADM TO BATTLE RHYTHM

This visualization treats the repository as a doctrinal pipeline: raw observation enters as unstructured evidence, the staff organizes it into structured operational data, and battle rhythm turns that into analyzed outputs, guidance, assessments, and orders. The design goal is lineage from source evidence to decision.

Unstructured Data

Raw Evidence Layer

Everything observed or received before it has been normalized into a common operational data model. This is the repository’s intake layer and should preserve original context, source provenance, and time.

Repository Rule

Preserve the original artifact, source, timestamp, classification handling, and link to the battle rhythm event or staff function that first used it.

Repository Rule

Every structured record should answer who, what, where, when, status, relationship, and source. This is the layer that feeds staff running estimates and battle rhythm event inputs.

Repository Rule

Every analyzed product should link backward to the structured operational records and raw evidence that produced it, and forward to the battle rhythm event, decision, or order it informed.

Battle Rhythm Minimum Feed

If the repository is supporting current operations, this is the recurring baseline data feed battle rhythm needs every cycle.

  • Running estimates
  • Updated situation template
  • Updated operational graphics and friendly locations
  • Significant activities in the last 24 hours
  • CCIR status
  • Decision support matrix
  • Collection plan and collection priorities
  • Target nominations
  • Previous-cycle BDA
  • Current HPTL / AGM / TSS
  • Protection, sustainment, IO, and CEMA assessments
  • Subordinate commander updates

Best Practical Rule

Use the doctrine-aligned split:

  • Unstructured = what was observed or received.
  • Structured = what the staff organized into a common operational data model.
  • Analyzed = what the staff assessed, recommended, approved, or decided.
Design intent: the repository should not just store files. It should let every commander's update, targeting board, assessment board, and working group answer the same four questions: what changed, why it matters, what decision is needed, and what source data supports that decision.
